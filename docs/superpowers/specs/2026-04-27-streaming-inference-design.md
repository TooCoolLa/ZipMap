# ZipMap 流式推理多线程架构设计

> **状态**: 已批准
> **日期**: 2026-04-27
> **目标**: 实现高性能、低延迟的 3D 重建推理流水线，通过多线程解耦图片加载、模型推理和结果保存。

## 1. 架构概览

系统采用生产者-消费者模型，分为三个核心阶段，通过线程安全的队列进行通信。

### 1.1 核心阶段
- **加载层 (Loading Layer)**:
  - **职责**: 从磁盘读取图片文件，进行预处理（缩放、归一化、填充），转换为 Tensor。
  - **线程数**: 默认 2。
  - **输出**: 图片 Tensor 及其元数据，存入 `image_queue`。
- **推理层 (Inference Layer)**:
  - **职责**: 从 `image_queue` 获取 Tensor，在 GPU 上运行 ZipMap 模型。
  - **线程数**: 1 (主线程/CUDA 串行)。
  - **输出**: 预测结果（深度图、位姿等），存入 `save_queue`。
- **保存层 (Saving Layer)**:
  - **职责**: 从 `save_queue` 获取预测结果，将其转换为 NumPy 格式并保存为压缩的 `.npz` 文件。
  - **线程数**: 默认 `os.cpu_count() - 2`。
  - **输出**: 磁盘上的单帧 `.npz` 文件。

### 1.2 队列配置
- **图片队列 (image_queue)**: 容量 200。防止加载速度远快于推理速度时消耗过多内存。
- **保存队列 (save_queue)**: 容量 80。防止推理速度远快于保存速度（通常发生在 I/O 密集型操作时）。

## 2. 数据流与格式

### 2.1 输入数据
- 支持常见的图片格式（.jpg, .png）。
- 按照文件名排序后进行流式处理。

### 2.2 推理结果 (.npz)
每帧预测结果保存为一个独立的 `.npz` 文件。包含但不限于：
- `pose_enc`: 位姿编码。
- `depth`: 深度图。
- `depth_conf`: 深度置信度。
- `extrinsic`: 相机外参。
- `intrinsic`: 相机内参。
- `images`: 原始图片数据。

### 2.3 可视化回看
重构可视化逻辑，使其能够遍历目标目录中的 `.npz` 文件，按需动态加载单帧数据，而非一次性加载全序列。

## 3. 技术实现细节

### 3.1 线程安全
- 使用 `queue.Queue` 进行线程间通信。
- 使用 `threading.Event` 控制各线程的停止和错误处理。

### 3.2 资源管理
- 自动检测 CPU 核心数以优化保存线程。
- 确保推理结束后，保存队列中的所有任务都被处理完再关闭保存线程。

## 4. 接口变更 (Gradio)
- 增加参数控制：`loading_threads`, `img_queue_size`, `save_threads`, `save_queue_size`。
- 更新后台处理函数，由原来的单循环改为启动多线程池。

## 5. 测试与验证
- **功能测试**: 验证保存的 `.npz` 文件是否可以被正常读取和渲染。
- **性能测试**: 比较多线程与单线程在长序列（>500帧）下的总耗时和峰值内存。
- **稳定性测试**: 模拟加载异常（文件损坏）时的系统恢复能力。
